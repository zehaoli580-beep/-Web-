## 一、GitHub 仓库管理方案

### 1.1 仓库初始化流程（组长操作）

```bash
# 1. 在 GitHub 上创建新仓库（设为 private）
#    仓库名：library-management-system

# 2. 在本地初始化项目
mkdir library-management-system
cd library-management-system
git init

# 3. 创建 .gitignore 文件（忽略 node_modules、.env、dist 等）
echo "node_modules/" >> .gitignore
echo ".env" >> .gitignore
echo "dist/" >> .gitignore

# 4. 创建初始目录结构
mkdir client server

# 5. 提交并推送到远程
git add .
git commit -m "init: 项目初始化"
git branch -M main
git remote add origin https://github.com/你的用户名/library-management-system.git
git push -u origin main
```

### 1.2 添加协作者

- 组长在 GitHub 仓库页面 → Settings → Collaborators → 添加组员的 GitHub 账号
- 两位组员接受邀请后，各自 clone 仓库：`git clone https://github.com/用户名/library-management-system.git`

### 1.3 分支策略

| 分支名 | 用途 | 谁维护 |
|:-------|:-----|:-------|
| `main` | 稳定版本，只合并测试通过的代码 | 组长审核合并 |
| `dev` | 开发主分支，三人各自从 dev 拉取功能分支 | 组长维护 |
| `feature/a-用户模块` | 组员A：用户模块 + 系统管理 | 组员A |
| `feature/b-图书模块` | 组员B：图书模块 + 分类管理 | 组员B |
| `feature/c-借阅模块` | 组员C：借阅模块 + 统计模块 | 组员C |

### 1.4 协作流程

```
1. 每天开始前：git checkout dev && git pull origin dev
2. 从 dev 拉取自己的功能分支：git checkout -b feature/a-用户模块
3. 写代码，频繁 commit
4. 功能完成后：git push origin feature/a-用户模块
5. 在 GitHub 上发起 Pull Request → dev
6. 组长审核后合并
7. 所有人同步：git checkout dev && git pull origin dev
```

> ⚠️ 重要规则：**不要直接在 main 或 dev 分支上写代码**，每个人在自己的功能分支上工作。每天至少 push 一次，防止本地代码丢失。

## 四、每日 Git 操作规范

```
每天开始：
  git checkout dev
  git pull origin dev
  git checkout -b feature/xxx-功能名   # 如果还没有自己的分支
  或：
  git merge dev                        # 在自己的分支上同步 dev 的更新

开发中：
  git add .
  git commit -m "feat: 完成了XX功能"
  每完成一个功能点就 commit 一次，不要攒到晚上一起 commit

每天结束：
  git push origin feature/xxx-功能名   # 推送到远程自己的分支
  如果有完整功能可以在 GitHub 上发起 PR → dev
```

### Commit 信息规范

| 类型 | 示例 |

## 五、风险与应对

| 风险 | 概率 | 应对方案 |
|:----|:----:|:---------|
| 某功能 AI 生成的代码跑不通 | 中 | 把报错复制给 AI 让它修；或者换简单的实现方式 |
| 合并代码时冲突 | 高 | 提前约定好各人改不同的文件；冲突时三人一起看 |
| 某组员进度落后 | 中 | 其他两人完成后帮忙；优先保证核心功能（借还书）能跑 |
| MongoDB 连接失败 | 低 | 检查 MongoDB 服务是否启动：`net start MongoDB` |
| 电脑出问题/代码丢失 | 低 | 每天 push，GitHub 上永远有最新代码 |
