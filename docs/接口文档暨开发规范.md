## 📐 二、通用规范（所有代码必须遵守）

### 2.1 命名规范

| 项目 | 规范 | 示例 |
|------|------|------|
| 数据库字段 | camelCase | `userId`, `createdAt`, `borrowDate` |
| API 路径 | 全小写 + 连字符 | `/api/v1/borrow-records` |
| Vue 组件 | PascalCase | `BookCard.vue`, `UserProfile.vue` |
| Vue 文件 | PascalCase | `LoginPage.vue`, `BookDetail.vue` |
| JS 变量/函数 | camelCase | `getBookList()`, `currentPage` |
| 路由 path | 全小写 + 连字符 | `/book-detail/:id` |
| 集合名(数据库) | 全小写复数 | `users`, `books`, `records` |

### 2.2 API 通用约定

- **基础路径**：`/api/v1`
- **请求体格式**：`Content-Type: application/json`
- **响应格式**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": { ... }
  }
  ```
- **分页请求参数**：`?page=1&pageSize=20`
- **分页响应格式**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {
      "list": [...],
      "total": 100,
      "page": 1,
      "pageSize": 20
    }
  }
  ```
- **日期格式**：ISO 8601 字符串，如 `"2026-06-08T10:30:00.000Z"`
- **认证方式**：`Authorization: Bearer <token>`
- **错误码范围**：
  | code | 含义 |
  |:----:|------|
  | 200 | 成功 |
  | 201 | 创建成功 |
  | 400 | 请求参数错误 |
  | 401 | 未登录 / Token 失效 |
  | 403 | 无权限 |
  | 404 | 资源不存在 |
  | 409 | 冲突（如重复预约） |
  | 500 | 服务器内部错误 |

### 2.3 前后端对接规则

- 前端所有请求通过 `client/src/api/` 下的 axios 封装发起
- 后端每个路由对应一个 controller 方法
- 前端调用时：`import { getBookList } from '@/api/book'`
- Token 统一在 axios 拦截器中自动携带（无需每个请求手动加）

## 🗄️ 三、数据库集合定义（MongoDB / Mongoose Schema）

### 3.1 users 集合

| 字段 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `_id` | ObjectId | 自动 | MongoDB 自动生成 |
| `username` | String | ✅ | 学号/工号，唯一索引 |
| `password` | String | ✅ | bcrypt 加密后存储 |
| `name` | String | ✅ | 真实姓名 |
| `role` | String | ✅ | `"student"` / `"teacher"` / `"admin"` |
| `college` | String | ✅ | 所属学院 |
| `phone` | String | | 手机号 |
| `email` | String | | 邮箱 |
| `status` | String | ✅ | `"active"` / `"disabled"`，默认 `"active"` |
| `borrowCount` | Number | ✅ | 当前借阅数量，默认 0 |
| `createdAt` | Date | 自动 | `timestamps` 自动生成 |
| `updatedAt` | Date | 自动 | `timestamps` 自动生成 |

> 注意：**角色不动态管理**。系统预设一个管理员账号（username: `admin`，role: `admin`），其余注册用户均为 `student`/`teacher`，不做前端动态权限配置。

### 3.2 books 集合

| 字段 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `_id` | ObjectId | 自动 | |
| `isbn` | String | ✅ | 国际标准书号，唯一索引 |
| `title` | String | ✅ | 书名 |
| `author` | String | ✅ | 作者 |
| `publisher` | String | ✅ | 出版社 |
| `year` | Number | ✅ | 出版年份 |
| `category` | String | ✅ | 分类名称（关联 categories 集合） |
| `total` | Number | ✅ | 馆藏总数 |
| `available` | Number | ✅ | 可借数量，默认 = total |
| `location` | String | ✅ | 馆藏位置，如 "3楼A区-12架" |
| `cover` | String | | 封面图片 URL（非上传文件，用户填写网络链接） |
| `description` | String | | 图书简介 |
| `status` | String | ✅ | `"available"` / `"borrowed"` / `"removed"`，默认 `"available"` |
| `borrowTimes` | Number | ✅ | 借阅次数，默认 0 |
| `createdAt` | Date | 自动 | |
| `updatedAt` | Date | 自动 | |

### 3.3 records 集合

| 字段 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `_id` | ObjectId | 自动 | |
| `userId` | ObjectId | ✅ | 关联 users 表 |
| `bookId` | ObjectId | ✅ | 关联 books 表 |
| `borrowDate` | Date | ✅ | 借书日期 |
| `dueDate` | Date | ✅ | 应还日期（借书日 + 30天） |
| `returnDate` | Date | | 实际归还日期 |
| `status` | String | ✅ | `"borrowed"` / `"returned"` / `"overdue"` / `"reserved"` |
| `renewed` | Boolean | ✅ | 是否已续借，默认 false |
| `fine` | Number | | 罚款金额，默认 0 |
| `reserveOrder` | Number | | 预约排队序号（仅 status=reserved 时有效） |
| `notifiedAt` | Date | | 预约到书通知时间 |
| `createdAt` | Date | 自动 | |
| `updatedAt` | Date | 自动 | |

> 索引建议：`{ userId: 1, status: 1 }`、`{ bookId: 1, status: 1 }`

### 3.4 categories 集合

| 字段 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `_id` | ObjectId | 自动 | |
| `name` | String | ✅ | 分类名称，如 "文学"、"科学"，唯一 |
| `createdAt` | Date | 自动 | |

### 3.5 configs 集合（系统参数）

| 字段 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `_id` | ObjectId | 自动 | |
| `key` | String | ✅ | 参数键名，唯一 |
| `value` | Mixed | ✅ | 参数值 |
| `description` | String | | 参数说明 |

预设数据：

| key | value | 说明 |

| `finePerDay` | 0.1 | 逾期每天罚款（元） |
| `reserveExpireHours` | 24 | 预约到书后保留时间（小时） |

### 3.6 messages 集合

| 字段 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `_id` | ObjectId | 自动 | |
| `userId` | ObjectId | ✅ | 接收消息的用户，关联 users 表 |
| `type` | String | ✅ | `"reserve_notify"` / `"overdue_remind"` / `"due_remind"` |
| `title` | String | ✅ | 消息标题 |
| `content` | String | ✅ | 消息正文 |
| `isRead` | Boolean | ✅ | 是否已读，默认 false |
| `relatedBookId` | ObjectId | | 关联的图书（可选） |
| `relatedRecordId` | ObjectId | | 关联的借阅记录（可选） |
| `createdAt` | Date | 自动 | |

> 索引建议：`{ userId: 1, isRead: 1 }`

后端逻辑（顺序执行）：
1. 检查用户 `borrowCount < maxBorrow`（从 configs 表读取）
2. 检查图书 `available > 0` 且 `status === "available"`
3. 图书 `available - 1`
4. 用户 `borrowCount + 1`
5. 创建记录：`{ userId, bookId, borrowDate: now, dueDate: now+30天, status: "borrowed" }`

成功响应（201）：
```json
{
  "code": 201,
  "message": "借书成功",
  "data": {
    "id": "recordId",
    "bookTitle": "深入理解计算机系统",
    "borrowDate": "2026-06-08T10:00:00.000Z",
    "dueDate": "2026-07-08T10:00:00.000Z"
  }
}
```

> 错误示例：图书已全部借出
```json
{
  "code": 409,
  "message": "该图书已全部借出",
  "data": null
}
```

#### 4.3.2 确认归还（需管理员）

```
PUT /api/v1/records/:id/return
```

后端逻辑（顺序执行）：
1. 找到 records 表中 id 对应的记录
2. 若 `returnDate` 已有值，返回错误（已归还）
3. 设置 `returnDate = now`
4. 计算是否逾期：`now > dueDate` → `status = "overdue"`，`fine = 天数差 × finePerDay`；否则 `status = "returned"`
5. 对应图书 `available + 1`
6. 用户 `borrowCount - 1`
7. 检查是否有预约队列 → 取 `reserveOrder` 最小的预约记录 → 向该用户发送"预约到书通知"消息（messages 集合中创建一条 `reserve_notify` 类型消息）

成功响应（200）：
```json
{
  "code": 200,
  "message": "归还成功",
  "data": {
    "fine": 0,
    "status": "returned"
  }
}
```

逾期情况：
```json
{
  "code": 200,
  "message": "归还成功，已逾期5天，罚款0.5元",
  "data": {
    "fine": 0.5,
    "status": "overdue"
  }
}
```

#### 4.3.3 续借（需登录）

```
PUT /api/v1/records/:id/renew
```

后端逻辑（顺序执行）：
1. 检查记录属于当前用户
2. 检查 `status === "borrowed"`
3. 检查 `renewed === false`
4. 检查当前日期是否在 `dueDate` 之前（逾期不可续借）
5. 检查该图书是否被其他人预约（有预约则不可续借）
6. `dueDate + 15天`，`renewed = true`

### 4.7 消息生成规则（内部逻辑，非 API）

以下逻辑不对外暴露接口，由后端在业务操作中自动触发：

**场景一：预约到书通知**
- 触发时机：管理员确认归还时
- 逻辑：还书后检查该书的预约队列 → 取 `reserveOrder` 最小的预约记录 → 生成一条 `reserve_notify` 类型消息 → 更新 `notifiedAt` 字段

**场景二：逾期提醒**
- 触发时机：用户登录时
- 逻辑：查询该用户 `status=borrowed` 且 `dueDate < now` 的记录 → 对每本逾期图书生成一条 `overdue_remind` 消息（仅当未生成过时）

**场景三：到期提醒**
- 触发时机：用户登录时
- 逻辑：查询该用户 `status=borrowed` 且 `dueDate` 在 3 天内且 `dueDate > now` 的记录 → 生成一条 `due_remind` 消息（仅当未生成过时）
